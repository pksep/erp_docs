# Техническое описание: Логика смены оборудования и ответственного пользователя

## 1. Обзор

Данный документ описывает процессы переназначения ответственных ресурсов для производственных операций:

- Переназначение пользователя (сборщика) через метод `setResponsibleUserToProductionOperation`.
- Переназначение оборудования (станка) через метод `setEquipmentToOperationPos`.

Оба метода находятся в `ProductionTaskService` и реализуют логику не только замены ID сущности в базе, но и поддержания целостности очереди задач и пересчета расписания (`start_time`) для участвующих сторон.

---

## 2. Логика смены оборудования (`setEquipmentToOperationPos`)

Метод вызывается при необходимости перевесить операцию металлообработки на другой станок (или снять её со станка).

### Алгоритм работы:

1. **Транзакция и валидация:** Открывается транзакция БД. Проверяется существование операции (`OperationPosition`) и, если передан `equipmentId` (не null), проверяется существование нового оборудования.
2. **Обновление записи:** Поле `equipment_id` в таблице операций обновляется на новое значение.
3. **Снятие оборудования (если передан `null`):** Если оборудование сбрасывается, у операции обнуляется время начала (`start_time = null`).
4. **Пересчет для старого оборудования:** Если у операции ранее был назначен станок (`oldEquipmentId`), система запускает функцию `recalculateStartTimeDetalByEquipment` для него. Это позволяет "схлопнуть" дыру в расписании старого станка и сдвинуть оставшиеся за ним задачи влево.
5. **Инициализация и пересчет для нового оборудования:**
   - Если передано новое оборудование, система проверяет наличие для него базовой точки отсчета времени (`StartDetalTime`).
   - Если точки нет, она создается с текущего времени (`calculateEndDate(new Date(), 0)`).
   - Запускается пересчет расписания `recalculateStartTimeDetalByEquipment` для нового станка, чтобы встроить переданную операцию в его очередь.

---

## 3. Логика смены пользователя (`setResponsibleUserToProductionOperation`)

Метод используется для передачи задачи (преимущественно сборочной) другому сотруднику.

### Алгоритм работы:

1. **Транзакция и валидация:** Открывается транзакция. Загружается `OperationPosition` вместе с родительской `ProductionOperationPos` (для проверки типа цеха). Запрашиваются данные нового пользователя через `UserService`.
2. **Перемещение в конец очереди:** Система находит максимальный порядковый индекс (`idx`) задач у нового пользователя (`max('idx')`). Передаваемой задаче назначается индекс `maxUserIndex + 1`. Таким образом, новая задача всегда падает **в конец списка** задач нового сотрудника.
3. **Обновление записи:** Поля `user_id` и `idx` обновляются в базе данных.
4. **Пересчет расписания (только для сборок - `StockOrderType.ass`):**
   - **Для старого пользователя:** Вызывается `recalculateStartTimeByUserId`, что позволяет пересчитать и уплотнить его оставшееся расписание.
   - **Для нового пользователя:** Проверяется наличие базового времени пользователя (`StartTime`). Если записи нет, она создается от текущего времени, с учётом рабочих часов (`calculateEndDate(new Date(), 0)`). Затем вызывается `recalculateStartTimeByUserId` для нового сотрудника для окончательного расчёта поля `start_time` добавленной операции.
5. Сброс состояния и применение изменений через `transaction.commit()`.

---

## 4. Архитектурные выводы и влияние на систему

- **Безопасность данных:** Использование транзакций защищает систему от рассинхронизации расписания в случае ошибки на одном из этапов пересчета.
- **Очереди задач:** При смене пользователя задача падает в конец его очереди, а при смене станка — встраивается в его очередь по внутренней логике сортировки станков.
- **Изолированность пересчетов:** Изменение затрагивает расписание ровно двух ресурсов (старого и нового), не требуя пересчета всего завода.
