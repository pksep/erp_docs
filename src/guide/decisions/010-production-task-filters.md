# Техническое описание: Логика работы фильтров в сервисе Производственных Заданий (ПЗ)

## 1. Обзор

Данный документ описывает механизмы фильтрации производственных операций при получении списка задач на производство (на примере метода `getPlanForProductionTask`). Фильтрация позволяет гибко отбирать ПЗ на основе родительских элементов, дочерних зависимостей, а также конкретных заказов (заказы покупателей и заказы на склад).

---

## 2. Описание основных фильтров

### Фильтр 1: Поиск по дочерним ПЗ (`findChildrensByProductionTask`)

Используется для получения родительских компонентов на основе массива ID производственных заданий.

**Алгоритм работы:**

1. По переданным ID дочерних ПЗ (`childrenByProductionTaskIds`) выполняются запросы в БД.
2. Извлекаются все связанные детали (для металлообработки) и сборки/изделия (для сборочных ПЗ), которые входят в эти дочерние ПЗ.
3. Собранные идентификаторы (`childDetalIds`, `childCbedIds`, `childProductIds`) пересекаются с уже существующим фильтром родительских элементов `byParents` (если он передан) через функцию валидации `intersect`.
4. Метод возвращает обновленный объект `byParents`, содержащий строго те узлы и детали, которые связаны с переданными дочерними ПЗ (или их пересечение с изначальным фильтром).

**Бизнес-смысл:** Позволяет отфильтровать общее дерево производственных задач так, чтобы отображались только те "родители" (сборки/изделия), готовность которых зависит от выбранных дочерних элементов (например, если в производстве выбрана конкретная партия деталей в металлообработке, можно узнать, к каким сборкам они относятся).

---

### Фильтр 2: Фильтрация по родительским элементам (`dto.byParents`)

Определяет выборку дочерних ПЗ, необходимых для обеспечения комплектующими конкретных родительских изделий, сборок или деталей.

**Алгоритм работы:**

1. Если в фильтре переданы `productIds` или `cbedIds`, система обращается к графовой базе данных Neo4j (метод `neo4jSpetificationService.getFullChildrens`).
2. Рекурсивно (на всю глубину спецификации) разворачивается дерево и собираются все входящие в него `cbedIds` и `detalIds`.
3. Одиночные детали (`detalIds` из фильтра) также добавляются к общему списку.
4. Все собранные идентификаторы агрегируются в объекте `filterByOnly`.
5. Итоговая выборка ПЗ ограничивается только теми операциями, которые привязаны к компонентам из этого собранного списка.

**Бизнес-смысл:** Поиск ответа на вопрос: «Какие именно детали и сборки (и связанные с ними работы) нужно произвести, чтобы собрать вот этот конкретный родительский узел (или изделие)?».

---

### Фильтр 3: Фильтрация по заказам склада (`isFilteringByStockOrder`)

Параметр срабатывает, когда источником потребности выбран "Склад" (в коде `dto.byOrder.customer === OrderTypeEnum.sclad` и передан массив `orderId`).

**Алгоритм работы:**

1. По переданным ID из БД загружаются складские заказы (`StockOrder`), включая вложенные в них позиции (`StockOrderItem`).
2. Из позиций собираются списки требуемых сборок (`cbedIds`), деталей (`detalIds`) и изделий (`productIds`).
3. Эти идентификаторы переносятся в фильтр `filterByOnly` (разделяясь по типу цеха: металлообработка или сборка).
4. SQL-запрос отсекает все производственные задания, кроме тех, компоненты которых явно фигурируют в запрошенных складских заказах.

**Бизнес-смысл:** Фильтрация плана производства исключительно под пополнение складских запасов (по конкретным заявкам на склад, игнорируя общие заказы покупателей).

---

### Фильтр 4: Фильтрация по заказам (дефициту) (`entityFilteringByShipment`)

Активируется при выборе источником потребности заказов покупателей (в коде `OrderTypeEnum.buyer`). Призван отсеивать элементы, по которым нет реальной потребности в производстве.

**Алгоритм работы:**

1. Сначала из базы загружаются компоненты (Сборки/Изделия или Детали), фигурирующие в выбранных заказах на отгрузку (`Shipments`).
2. Затем полученный пул компонентов пропускается через метод `entityFilteringByShipment` сервиса `ShipmentsService`.
3. Внутри запускается ресурсоемкая проверка **реального дефицита** (`formatinglistShipmentWithCount`): система анализирует, действительно ли данный компонент всё ещё требуется для конкретного `shipmentId`, отсеивая те ПЗ, потребности по которым уже закрыты текущими остатками на складе или завершенными ПЗ.
4. Оставшиеся (дефицитные) идентификаторы возвращаются и добавляются в ограничивающий пул `filterByOnly` с применением пересечения (`intersect`), если активны другие фильтры.

**Бизнес-смысл:** Позволяет отобразить на доске производства **только те задачи, которые физически необходимы** для закрытия выбранных заказов покупателей прямо сейчас (исключая то, что уже произведено, собрано или имеется на складе сверх нормы).

---

## 3. Синергия фильтров (Комбинирование)

Архитектурно фильтры спроектированы так, чтобы работать совместно. При активации сразу нескольких условий применяется логическое **«И»** с помощью утилитной функции пересечения множеств `intersect`.

**Пример сценария:**
Если пользователь выбирает конкретный заказ покупателя (Фильтр 4) И конкретную крупную сборку (Фильтр 2), система:

1. Развернет сборку через Neo4j (соберет все её дочерние детали).
2. Выявит дефицит по выбранному заказу покупателя через `entityFilteringByShipment`.
3. Пересечет оба списка и покажет план-график **только** на те дочерние детали выбранной сборки, по которым в данный момент числится дефицит в рамках целевого заказa.
