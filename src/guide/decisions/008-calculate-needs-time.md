# Техническое описание: Логика расчета "Расчетной даты изготовления" (calculateNeedsTime)

## 1. Обзор

Показатель **calculateNeedsTime** предназначен для расчетной даты и показывает реальную готовность сущности, по которым зафиксирован дефицит. Система использует два типа расчета в зависимости от бизнес-контекста:

1.  **Тип 1 (Дефицитный):** Расчет даты на основе ПЗ, которая **полностью покрывает** дефицит по родительской сущности.
2.  **Тип 2 (Проекционный):** Расчет даты по **самой ближайшей** доступной ПЗ в графике.

---

## 2. Общие принципы для обоих типов

- **Фокус на финал:** Оба метода анализируют только **последние операции технологического процесса** в технологической цепочке (максимальный `idx`), так как деталь считается готовой для сборки только после прохождения финального этапа металлообработки.
- **Опорная точка:** Расчет всегда отталкивается от `start_time` (времени начала работы) в графике производства.
- **Производственный календарь:** Итоговая дата вычисляется через функцию `calculateEndDate`, которая учитывает рабочие смены и графики.

---

## 3. Детальное описание типов расчета

### Тип 1: Дефицитный расчет (Покрытие потребности)

Используется в модулях планирования дефицитов для определения реального срока, когда сборка сможет получить _необходимое_ количество деталей.

**Алгоритм:**

1.  **Сбор данных:** Метод ` getLastsOperationPositions` находит последние операции всех активных ПЗ для данной детали.
2.  **Накопительный итог (Метод `setActualLastOperation`):**
    - Система получает сумму дефицита (например, -50 шт).
    - Итерирует список ПЗ, отсортированный по дате старта.
    - Складывает количества (`quantity`) выпускаемых деталей из каждой ПЗ.
    - **Целевая ПЗ** — это та задача, на которой накопленная сумма выпуска перекрывает сумму дефицита.
3.  **Расчет времени:**
    - Берется `start_time` целевой ПЗ.
    - Прибавляется время выполнения последней операции для объема этой ПЗ (`createTimeByLastOperation`).

**Бизнес-смысл:** Ответ на вопрос «Когда у меня будет достаточно деталей, чтобы закрыть весь дефицит по этой позиции?».

---

### Тип 2: Проекционный расчет (Ближайшая партия)

Используется для оперативного мониторинга и быстрой оценки состояния производства, когда важен сам факт ближайшего выхода продукции.

**Алгоритм:**

1.  **Сбор данных:** Метод `getOneLastOperationPosition` выполняет SQL-запрос с `LIMIT 1`.
2.  **Выбор ПЗ:**
    - Система находит финальные операции всех ПЗ.
    - Выбирает ту, которая имеет **самый ранний `start_time`** в графике.
3.  **Расчет времени:**
    - **calculatedCreateTime:** Берется готовое значение `plan_ready_time` из базы данных (установленное планировщиком).
    - **calculateNeedsTime:** Берется `start_time` и к нему прибавляется расчетное время по техпроцессу для всей партии (`operationCreateTime`).

**Бизнес-смысл:** Ответ на вопрос «Когда выйдет самая первая партия деталей, независимо от того, сколько мне их нужно всего?».

---

## 4. Сравнительная таблица

| Параметр           | Тип 1 (Дефицитный)                                | Тип 2 (Проекционный)                                   |
| :----------------- | :------------------------------------------------ | :----------------------------------------------------- |
| **SQL запрос**     | ` getLastsOperationPositions` (возвращает список) | `getOneLastOperationPosition` (возвращает одну запись) |
| **Логика подбора** | Перебор ПЗ до покрытия суммы дефицита             | Выбор самой ранней по `start_time`                     |
| **Объем партии**   | Количество конкретной «покрывающей» ПЗ            | Общий объем заказов по ПЗ (`totalShipment`)            |
| **Точность**       | Высокая (учитывает объем потребности)             | Оптимистичная (учитывает только скорость)              |
| **Основное поле**  | `calculateNeedsTime`                              | `calculateNeedsTime` + `calculatedCreateTime`          |

---

## 5. Триггеры пересчета

Оба типа дат подлежат обновлению при следующих событиях:

1.  Изменение `start_time` любой связанной операции.
2.  Изменение спецификации сборки (меняется дефицит).
3.  Архивация или создание новых производственных задач.
4.  Изменения количества детали на складе.
