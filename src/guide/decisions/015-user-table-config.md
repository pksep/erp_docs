# Техническое описание: Конфигурация таблиц пользователей (`UserTableConfig`)

## 1. Обзор

Данный документ описывает архитектуру и реализацию системы хранения персональных настроек отображения таблиц для пользователей.
Основная цель — позволить пользователям скрывать или отображать колонки в таблицах, сохраняя эти настройки между сессиями.

---

## 2. Архитектура

### Модель данных

Создана новая модель `UserTableConfig`, которая имеет связь "один-к-одному" с моделью `User`.

- **Таблица**: `user_table_configs`
- **Поля**:
  - `id`: Primary Key.
  - `user_id`: Foreign Key (уникальный), ссылка на `users(id)`. Каскадное удаление.
  - `metalworking`: Массив объектов `JSONB`. Каждый объект содержит `columnName` (строка) и `isShow` (boolean).

### Валидация (Zod)

Для обеспечения целостности данных на уровне приложения используются схемы Zod в пакете `zod-shared`:

- `UserTableConfigItemSchema`: Валидирует отдельный элемент конфигурации колонки.
- `UserTableConfigSchema`: Валидирует всю модель конфигурации.
- Используется `z.nativeEnum(tablesEnumConfig)` для ограничения допустимых имен колонок.

---

## 3. Автоматизация и Жизненный цикл

### Создание (Hooks)

В модели `User` реализован хук `afterCreate`:

- При создании нового пользователя автоматически создается запись в `UserTableConfig` с пустыми настройками.
- Это гарантирует наличие объекта конфигурации при первом обращении пользователя к настройкам.

### Миграция и Сидирование

Миграция `20260316091000-create-user-table-config.js` выполняет следующие действия:

1. Создание таблицы со всеми ограничениями (FK, UNIQUE).
2. Выборка всех существующих пользователей.
3. Сидирование (заполнение) таблицы `user_table_configs` для всех текущих пользователей.

---

## 4. Управление конфигурацией (`updateTableConfig`)

Логика обновления реализована в `UsersService.updateTableConfig`.

### Процесс обновления:

1. Принимается DTO `UpdateUserTableConfigDto`, содержащее `userId`, `configKey` и `newConfigObject`.
2. Поиск существующей конфигурации.
3. **Безопасная актуализация**: Использование функции `createCorrectColumnConfig` из `zod-shared`.
   - Данная функция сопоставляет присланный массив с эталонным перечнем колонок (`metalloworkingTableConfig`).
   - Новые колонки, отсутствующие в присланном объекте, добавляются со значением `isShow: true`.
   - Лишние ключи отсекаются.
   - Это защищает систему от поломки при добавлении новых колонок в код без необходимости ручного обновления всех записей в БД.
4. Сохранение обновленного массива в соответствующее поле модели.

---

## 5. Расширяемость

Архитектура спроектирована с учетом будущих расширений:

- Поле `configKey` в методе обновления позволяет добавлять настройки для других таблиц (сборка, склады и т.д.) путем добавления новых полей в модель `UserTableConfig` и расширения `switch` в сервисе.
- Типизация через `zod-shared` обеспечивает консистентность между фронтендом и бэкендом.
