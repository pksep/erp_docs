# Техническое описание: Логика архивирования производственного задания (`banProductionTask`)

## 1. Обзор

Данный документ описывает процесс архивирования (удаления из активного рабочего процесса) Производственного задания (ПЗ). Основная логика реализована в методе `banProductionTask` сервиса `ProductionTaskService`.

Процесс архивации — это не просто смена статуса одной записи, а комплексная каскадная операция, включающая архивацию всех вложенных позиций (узлов/деталей), снятие технологических операций с исполнителей и пересчет расписания (календарей) освободившихся производственных ресурсов (пользователей и станков).

---

## 2. Точки входа

В системе существуют два основных пути инициации архивации ПЗ:

1. **Событийная модель (`banProductionTaskEvent`):** Метод создает запись о действии в `ActionsService` (логирование) и отправляет событие `ArchiveProductionTaskEvent` в очередь (BullMQ / RabbitMQ). Обработчик очереди (`ArchiveProductionTask`) подхватывает задачу и вызывает основной метод `banProductionTask`.
2. **Прямой вызов (`banProductionTask`):** Ядро логики. Метод может принимать готовую транзакцию или создавать новую для обеспечения целостности (`ACID`).

---

## 3. Алгоритм работы `banProductionTask`

### Шаг 1: Загрузка данных и транзакционность

Система резервирует транзакцию базы данных. Загружается целевое ПЗ вместе с его неархивированными производственными позициями (`ProductionOperationPos`) и конкретными технологическими операциями (`OperationPosition`). Если ПЗ не найдено, выбрасывается исключение.

### Шаг 2: Изменение статуса корневого ПЗ

Корневой записи Производственного задания присваивается флаг `ban = true`. Изменения фиксируются в БД.

### Шаг 3: Каскадная архивация позиций (`banProductionOperationPos`)

Система итерируется по всем активным позициям ПЗ и для каждой вызывает дочерний метод архивации `banProductionOperationPos`.
Логика внутри `banProductionOperationPos`:

- Устанавливается `ban = true` для родительской позиции (`ProductionOperationPos`).
- Вызывается актуализация индексов (`actualOperationPosIndexes`) для родительского ПЗ.
- Массово устанавливается `ban = true` для всех вложенных `OperationPosition` (непосредственно задания исполнителям).
- **Локальный пересчет:** Если позиция относится к сборке (`StockOrderType.ass`), система собирает всех участвовавших пользователей (`user_id`) и сразу пересчитывает для них время начала (`recalculateStartTimeByUserId`).
- Отправляется событие об изменении связей потребности (`setActualTaskRelative`).

### Шаг 4: Глобальный сбор и пересчет расписания ресурсов

Во время прохода по позициям на Шаге 3, метод `banProductionTask` накапливает глобальные списки (множества `Set`) всех затронутых архивацией ресурсов:

- Уникальные исполнители массива сборок (`userIdsSet`).
- Уникальное оборудование/станки металлообработки (`equipmentIdsSet`).

После архивации всех сущностей система запускает цикл высвобождения расписания:

- **Для сборщиков (пользователей):** вызывается метод `actualOperationPos({ userId })`. Это перестраивает очередь их задач и "схлопывает" дыры в расписании, сдвигая оставшиеся невыполненные задачи на более раннее время (компенсируя удаленные операции ПЗ).
- **Для оборудования (станков):** вызывается `recalculateStartTimeDetalByEquipment({ equipmentId })`, который перестраивает очередь задач для конкретных металлообрабатывающих участков, выполняя ту же функцию сдвига времени влево.

### Шаг 5: Фиксация (Commit)

Если вся цепочка выполнена успешно, транзакция закрывается (`transaction.commit()`). В случае любой ошибки (например, сбоя при расчете времени) выполняется `transaction.rollback()`, и система возвращается в исходное состояние, обеспечивая консистентность расписания.

---

## 4. Бизнес-выводы

- Удаление ПЗ немедленно высвобождает ресурсы фабрики/цеха без необходимости ждать ночного пересчета планировщика.
- Благодаря фоновым процессам (BullMQ) тяжелая операция каскадного снятия флагов и пересчета расписаний на десятках исполнителей не блокирует пользовательский интерфейс.
