

# [ADR-0006] Расчёт текущей и складской загруженности сотрудников подразделения

- **Status:** Accepted
- **Date:** 2026-01-30
- **Members:** Михаил
- **Tags:** #architecture #production-tasks #workload-analysis #erp #resource-management

## Контекст

Для эффективного планирования производства и анализа текущей ситуации в цехах необходимо иметь сводную информацию по загрузке каждого сотрудника. Метод `getProductionTasksByAllUsers` агрегирует данные обо всех активных производственных задачах (ПЗ) и вычисляет метрики занятости для пользователей конкретного подразделения (`EnumSubdivision`).

Расчёт разделяется на два направления:

1. **Занятость сотрудника (`employment`):** Общий объём работ, закрепленный за пользователем во всех активных ПЗ/ЗС.
2. **Занятость на данный момент (`currentEmployment`):** Детализация по типам операций, доступных для выполнения прямо сейчас (с учётом готовности предыдущих этапов).

## Основные компоненты

### 1. Сбор и обработка данных (`production-task.service.ts`)

#### Этап 1: Инициализация

- Получение всех активных ПЗ (`ban: false`), отсортированных по `due_data` (срок сдачи).
- Получение списка пользователей целевого подразделения и инициализация структуры `IUnicalUsers`.

#### Этап 2: Параллельная обработка задач

Используется `PromisePoolService` (лимит 10 потоков). Для каждой ПЗ загружаются позиции операций и технологические процессы (Металлообработка или Сборка).

#### Этап 3: Алгоритм расчёта метрик

1. **Анализ дефицита склада:** Вычисляется `orderForStorage` (разница между заказом склада и уже созданным количеством).
2. **Учет отметок (Marks):** Через карту `marksQuantityInOperation` определяется остаток невыполненных работ по каждой конкретной операции.
3. **Очередность выполнения:**
   - Проверяется статус "Возможность начать": если предыдущая операция по техпроцессу имеет больше выполненных отметок, чем текущая, или если текущая операция — первая в цепочке.
4. **Расчёт времени:** Используется функция `worksHors`, которая переводит количество сущностей в нормо-часы на основании данных техпроцесса.

## Детализация метрик

### 1. Занятость сотрудника (`employment`)

Данный блок показывает общую нагрузку на пользователя по всем закрепленным за ним задачам.

- **Кол-во позиций (`quantityPositions` / `storageQuantityPositions`):** Количество уникальных позиций ПЗ/ЗС, в которых пользователь назначен ответственным.
- **Кол-во сущностей (`quantityEntities` / `storageQuantityEntities`):** Суммарное количество изделий (единиц продукции) во всех ПЗ/ЗС, где участвует пользователь.
- **Время ПЗ (`executionTime`):** Сумма времени, необходимого для закрытия текущих операций во всех ПЗ (расчёт: _время операции × остаток отметок до закрытия ПЗ_).
- **Время ЗС (`storageExecutionTime`):** Сумма времени, необходимого для закрытия потребностей склада (расчёт: _время операции × остаток отметок до закрытия ЗС_).

### 2. Занятость на данный момент (`currentEmployment`)

Группировка данных по **типу операции** (`tOperationId`), показывающая работу, которую сотрудник может выполнять в текущую секунду.

- **Кол-во позиций ЗС/ПЗ:** Количество уникальных заказов (складских или клиентских), по которым текущая операция доступна для работы.
- **Кол-во сущностей ЗС/ПЗ:** Количество единиц продукции, которые уже прошли предыдущий этап обработки и "ждут" обработки текущим пользователем.
- **Время ПЗ (`executionTime`):** Время, оставшееся для выполнения доступных (разблокированных предыдущими этапами) сущностей по текущим ПЗ.
- **Время ЗС (`storageExecutionTime`):** Время, необходимое для закрытия доступных складских сущностей.

## Бизнес-правила

1.  **Уникальность позиций:** Чтобы избежать завышения показателей, при подсчете `quantityPositions` используется связка `userId + entityId + productionTaskId`. Если у пользователя две разные операции в рамках одной позиции, позиция в общую занятость засчитывается один раз.
2.  **Логика "Оставшейся работы":** Система игнорирует выполненные отметки. В расчёт идет только дельта между требуемым количеством и фактически проставленным в системе.
3.  **Зависимость от техпроцесса:** Если предыдущая операция (по индексу `idx`) не выполнена, сущности не попадают в `currentEmployment` (занятость на данный момент), но учитываются в `employment` (общая занятость), так как работа за сотрудником закреплена.
4.  **Сортировка:** Итоговый отчет сортируется по `login` пользователя (на русском языке).

## Обоснование архитектурного решения

1.  **Производительность:** Использование `Promise.all` и `PromisePool` позволяет обрабатывать сотни ПЗ одновременно, не блокируя основной поток Node.js.
2.  **Масштабируемость:** Расчёт ведется через `Map`, что позволяет быстро агрегировать данные даже при большом количестве сотрудников в подразделении.
