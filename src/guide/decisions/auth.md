# Аутентификация и авторизация в ERP

В системе реализована JWT-аутентификация с использованием короткоживущих Access-токенов и долгоживущих Refresh-токенов в защищенных HttpOnly куках.

## 1. Архитектура

Система состоит из трех основных компонентов:

- **`AuthMiddleware`**: Перехватывает все входящие запросы, проверяет наличие и валидность токенов в куках.
- **`AuthService`**: Содержит бизнес-логику генерации токенов, их обновления, управления кэшем пользователей и охраны удаленного доступа.
- **`UsersService`**: Отвечает за хранение хэшей `refresh_token` в базе данных и управление состоянием пользователей (запрет/бан).

## 2. Жизненный цикл токенов

### Access Token (30 минут)
- Используется для авторизации каждого запроса.
- Содержит ID пользователя, роли, табель и флаг удаленного доступа.
- При истечении `AuthMiddleware` автоматически пытается выполнить бесшумное обновление (silent refresh).

### Refresh Token (24 часа)
- Используется только для получения новой пары токенов.
- Хэш от `refresh_token` (SHA-256) хранится в базе данных (`User.refresh_token`).
- **Ротация**: При каждом обновлении выпускается новая пара Access + Refresh токенов, а старый Refresh Token в БД заменяется новым. Это предотвращает повторное использование украденных токенов.

### Локальная разработка (`NODE_ENV === 'local'`)
- В режиме `local` токены выдаются со сроком жизни 100 лет (36500 дней), что исключает необходимость постоянного перезахода.

## 3. API Эндпоинты

| Метод | Путь | Описание |
| :--- | :--- | :--- |
| `POST` | `/api/auth/login` | Вход в систему, установка кук `access_token` и `refresh_token`. |
| `POST` | `/api/auth/refresh` | Принудительное обновление токенов. |
| `POST` | `/api/auth/logout` | Очистка сессии в БД/кэше и удаление кук. |
| `POST` | `/api/auth/check` | Технический эндпоинт для проверки валидности токена. |

## 4. Механизмы безопасности

- **HttpOnly & SameSite=Lax**: Токены недоступны для JavaScript на стороне клиента (защита от XSS) и защищены от определенных видов CSRF-атак.
- **Redis Caching**: Данные пользователя для Middleware кэшируются на 24 часа. При банне пользователя кэш немедленно инвалидируется.
- **Remote Work Guard**: Проверка на соответствие хоста (из заголовка `Host`/`X-Forwarded-Host`) разрешенным локальным сетям. Если запрос идет с внешнего хоста, проверяется флаг `user.remote_work`.
- **Compare-and-Swap (Update)**: Обновление `refresh_token` в БД происходит атомарно через условие `where refresh_token = oldHash`, что исключает гонки при одновременных запросах.

## 5. Исключения (Public Paths)

Некоторые пути доступны без авторизации (настроены в `AuthMiddleware.shouldSkipAuth`):
- `/api/auth/login`
- `/api/auth/refresh`
- `/api/users/list` (и его варианты с параметрами)
- Сервисные маршруты SSE (`/sse-`).

## 6. Логика блокировки пользователя

Если администратор блокирует пользователя (`ban = true`):
1. В `UsersService.userToArchive` устанавливается флаг в БД.
2. Немедленно вызывается `invalidateUserCache`, удаляющий запись из Redis.
3. При следующем запросе (даже по валидному Access-токену) `AuthMiddleware` не найдет пользователя в кэше, обратится в БД, увидит бан и прервет сессию.

## 7. Поведение токенов

1. Access Token
- Выдается на 30 минут
- Когда `access_token` истекает, middleware перехватывает событие
- Middleware автоматически пробует использовать `refresh_token` из кук
- Если `refresh_token` валиден, вызывается `authService.refreshTokens`, который:
    - проверяет хэш рефреш-токена в БД
    - генерирует новую пару токенов
    - устанавливает новые куки
- Для пользователя это происходит незаметно (`silent refresh`), сессия продолжается

2. Refresh Token
- Выдается на срок 24 часа. Это фактически является "таймаутом бездействия"
- Так как при каждом обновлении `access_token` (каждые 30 мин активной работы) выдается новый `refresh_token` со сроком еще на 24 часа, пользователь может работать бесконечно долго, пока он активен
- Если пользователь закроет вкладку и вернется через 25 часов:
    - `access_token` просрочен
    - `refresh_token` просрочен
    - middleware выбросит ошибку с сообщением "Сессия истекла, пожалуйста, войдите снова"
    - фронтенд перенаправит пользователя на страницу логина
