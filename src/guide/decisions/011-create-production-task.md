# Техническое описание: Логика создания производственного задания (`createProductionTask`)

## 1. Обзор

Данная документация описывает алгоритм работы метода `createProductionTask` в `ProductionTaskService`. Функция отвечает за формирование новых производственных заданий (ПЗ) для цехов (сборка или металлообработка) на основе выбранных пользователем позиций, контроль ограничений по количеству и логирование действий пользователя.

---

## 2. Общие принципы и валидация

### Транзакционность

Весь процесс создания ПЗ обернут в транзакцию базы данных (`sequelize.transaction()`). Это гарантирует, что в случае любой ошибки при прохождении цепочки валидаций или сохранении данных (например, отказ базы или ошибка валидации) ни одна запись не будет сохранена частично, и система вернется в исходное состояние (rollback).

### Валидация исполнителей и оборудования

Система жестко контролирует полноту данных перед созданием ПЗ:

- **Для всех типов ПЗ:** Проверяется обязательное наличие исполнителя (`employeeId`) для каждой технологической операции.
- **Для металлообработки (`StockOrderType.metall`):** Дополнительно требуется обязательное указание оборудования/станка (`equipmentId`) для каждой операции, так как станки имеют собственный календарь занятости.

### Лимиты на создание (защита от перепроизводства)

До момента сохранения ПЗ система вычисляет максимально допустимое к производству количество (дефицит):

- Производится подсчет по формуле: `kolvo_shipments` (сколько всего требуется по заказам склада) минус `kolvo_create` (сколько уже запущено в производство).
- Если запрашиваемое к производству количество детали/сборки + уже созданное в рамках других ПЗ количество превышает требуемое по заказам отгрузок (`countByShipments`), создание блокируется с ошибкой **«Превышено максимальное кол-во к созданию пз»**.

---

## 3. Этапы создания данных

### Шаг 1: Создание шапки ПЗ (`ProductionTask`)

В первую очередь в базе данных создается корневая запись Производственного задания (`ProductionTask`).
Сохраняются следующие поля:

- `due_data` (ожидаемая дата завершения или крайний срок, переданный с клиента).
- `working_type` (тип цеха: сборка или металлообработка).
- `details_filters` (сохраненные фильтры ПЗ).
- `description` (комментарий или описание к задаче).

### Шаг 2: Создание позиций производственных операций (`ProductionOperationPos`)

Для каждой переданной производственной сущности (детали или сборки) вызывается вспомогательный метод `createProductionOperationPos`.
Внутри этого метода:

- Создаются привязки операций конкретном ПЗ (`production_task_id = newProductionTask.id`).
- Позиции привязываются либо к `ass_id` (для сборок), либо к `metall_id` (для металлообработки).
- Сохраняются переданные операции, их индексы (`idx`) и заказанное количество (`quantity`). Этап также включает расчет `start_time` позиций в зависимости от привязанного оборудования или исполнителя.

### Шаг 3: Логирование (Действия и история)

Любое создание ПЗ оставляет след в истории (модуль `ActionsService`):

1. **Главныи лог:** Создается действие "Создал производственное задание" (привязка к типу сущности `ActionEntityTypes.production_task` и ID нового ПЗ). Возвращается `actionId`.
2. **Детализированный лог (Связи):** Запускается еще один цикл по всем позициям (сборкам/металлу). Для каждой создается дочернее действие ("Прикреплена к производственной задаче"), которое ссылается на главную операцию через `relativeActionId: actionId`.
3. **Обновление связей сущностей:** Вызывается встроенный метод Sequelize `$add`, чтобы добавить связь между созданным ПЗ и конкретной сборкой или операцией металлообработки на уровне таблиц пересечения.

---

## 4. Результат работы

После успешного выполнения всех шагов:

1. Выполняется `transaction.commit()`.
2. Метод возвращает объект созданного `ProductionTask`.
3. Задачи появляются на досках соответствующих структурных подразделений, а время начала операций (`start_time`) распределяется по назначенным ресурсам согласно их календарям загруженности.
